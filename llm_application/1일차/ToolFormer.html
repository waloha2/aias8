ToolFormer

LLM이 스스로 도구 사용법을 배우는 방법

Schick et al., Meta AI Research, 2023

핵심 아이디어 한 줄 요약

LLM이 자기 지도 학습(self-supervised)으로
텍스트 중간에 API 호출을 삽입하는 법을 배운다.

기존 접근법의 한계

  • 대량의 사람이 만든 도구 사용 데모 필요
  • 특정 도구에 종속된 학습
  • 모델 능력 자체를 제한 (도구에 의존)
  • 도구 사용 시점을 사람이 결정

ToolFormer의 혁신

  • 사람의 annotation 없이 자동 학습
  • 다양한 도구를 통합적으로 학습
  • 도구가 도움될 때만 선택적 사용
  • 모델이 스스로 언제 어떤 도구를 쓸지 결정

1. 왜 도구가 필요한가?

LLM의 근본적 한계들

🧮

산술 능력 부족

"593 × 472 = ?" 같은 단순 계산도 자주 틀림. 파라미터에 계산 로직이 정확히 인코딩되지 않음.

Q: 593 × 472 = ?
LLM: 279,896 ❌
정답: 279,896... 아, 맞을 때도 있지만
큰 수에서는 불안정!
📅

최신 정보 부재

학습 데이터 이후의 사건, 날짜, 사실을 모름. "오늘 날씨"나 "최근 뉴스"에 답할 수 없음.

Q: 현재 미국 대통령은?
LLM: (학습 시점 기준으로 답변)
→ 최신 정보 반영 불가
🌐

사실 확인 불가

Hallucination 문제 — 그럴듯하지만 틀린 정보를 자신있게 생성.

Q: 에펠탑의 높이는?
LLM: 324m (맞을 수도, 틀릴 수도)
→ 검증 수단이 없음
🔄

번역 품질 한계

저자원 언어(low-resource language) 번역에서 품질 저하. 전문 번역 API가 더 정확한 경우 많음.

Q: "안녕하세요"를 스와힐리어로?
LLM: (불확실한 번역)
번역 API: "Habari" ✓

해결책: 외부 도구 + LLM

사람이 계산기, 검색엔진, 달력을 쓰듯이 — LLM도 도구를 쓰면 됩니다!

피츠버그는 [QA("피츠버그의 설립 연도")] → 1758 년에 설립된 도시로,
현재까지 약 [Calculator(2023 - 1758)] → 265 년의 역사를 가지고 있습니다.

2. 사용하는 도구들 (5가지 API)

🔎

Question Answering (QA)

[QA("에펠탑을 설계한 사람은?")] → "귀스타브 에펠"

용도: 위키피디아 등의 지식 검색, 사실 확인

🧮

Calculator

[Calendar()] → "Today is February 9, 2023"

용도: "오늘", "내일", "지난주" 등 시간 관련 질문

🌐

Wikipedia Search

위키피디아 문서 검색 & 요약 반환

[WikiSearch("Transformer model")] → "The Transformer is a deep learning..."

용도: 구체적 사실, 배경 지식 조회

🌍

Machine Translation

200개 언어 지원 (NLLB 모델 기반)

[MT("Thank you", English, Korean)] → "감사합니다"

용도: 저자원 언어 번역, 정확한 번역 필요시

통합 API 호출 형식

e(api_name, input) → output

예시: e(Calculator, 593 * 472) → 279896
예시: e(QA, "수도 of France") → Paris

텍스트 내 표현:
... 텍스트 [api_name(input) → output] 텍스트 계속 ...

4. Step 1: API 호출 샘플링 (상세)

사람 없이 어떻게? — 핵심 아이디어

ToolFormer의 천재적인 점은 LLM 자기 자신을 이용해 학습 데이터를 만든다는 것입니다. 3단계로 나뉩니다:

A

Few-shot 프롬프트

"이런 식으로 API를 쓸 수 있어" 라는 예시를 몇 개 보여줌

→
B

위치별 확률 계산

텍스트의 모든 위치에서 "<API>" 토큰이 나올 확률 측정

→
C

후보 생성

확률 높은 위치에서 실제 API 호출 텍스트를 여러 개 생성

이 전체 과정에서 사람이 하는 일은 API별 프롬프트 예시 몇 개를 작성하는 것뿐입니다. (각 API당 2~5개) — 이것은 annotation이 아니라 API의 "사용법 설명서"에 해당합니다.

Step A: 실제 Few-shot 프롬프트 (논문에서 사용된 것)

각 API마다 "이 API는 이렇게 쓰는 거야"라는 설명 + 예시를 담은 프롬프트를 만듭니다. 탭을 클릭하여 각 API의 실제 프롬프트를 확인하세요.

왜 이게 "사람 개입 없음"인가요?

이 프롬프트는 API의 사용법만 알려줍니다. "이 텍스트의 이 위치에서 이 API를 써라"라고 지정하지 않습니다. 마치 사람에게 "계산기는 이렇게 쓰는 거야"라고 가르치는 것과 같습니다. 어디서 쓸지, 어떤 입력을 넣을지는 모델이 스스로 판단합니다.

Step B: 위치별 확률 계산 — 원리

프롬프트 + 원본 텍스트를 LLM에 넣고, 텍스트의 매 토큰 사이에서 다음 토큰이 <API> 일 확률을 구합니다.

텍스트 x = [x₁, x₂, ..., xₙ] 이 주어졌을 때,

각 위치 i (= 1, 2, ..., n) 에서:
p_i = P_M( <API> | P(e), x₁, x₂, ..., x_i )

여기서 P(e) = API 사용법을 알려주는 few-shot 프롬프트
P_M = 사전학습된 LLM (GPT-J 6.7B)

직관: "이 위치까지 읽었을 때, 모델이 API 호출을 시작하고 싶어하는 정도"

Deep Dive: GPT-J 내부에서 정확히 무슨 일이 일어나는가?

"확률을 계산한다"는 말이 추상적으로 느껴질 수 있습니다. Transformer의 내부 동작을 단계별로 시각화해 보겠습니다.

핵심: <API> 토큰은 어디서 왔는가?

GPT-J의 원래 vocabulary에는 <API> 토큰이 없습니다. ToolFormer가 특수 토큰을 추가합니다.

원래 GPT-J vocabulary

총 50,400개 토큰:

[0] "<|endoftext|>"
[1] "!"
[2] "\""
... (일반 텍스트 토큰들)
[262] " the"
[318] " a"
...
[50,399] (마지막 토큰)

<API> 토큰 없음!

ToolFormer가 확장한 vocabulary

총 50,400 + 2 = 50,402개 토큰:

[0] "<|endoftext|>"
... (기존 50,400개 그대로)
[50,399] (기존 마지막 토큰)

[50,400] "<API>" ← 새로 추가!
[50,401] "</API>" ← 새로 추가!

Embedding layer도 2행 추가됨

새 토큰의 임베딩은 어떻게 초기화하나?

랜덤 초기화 또는 기존 유사 토큰의 임베딩으로 초기화합니다. 처음에는 <API> 토큰의 확률이 매우 낮지만, few-shot 프롬프트에서 이 토큰이 사용되는 패턴을 보여주면 LLM의 in-context learning 능력으로 "이 토큰이 여기에 올 수 있겠다"고 판단할 수 있게 됩니다. Fine-tuning 단계에서 이 임베딩이 정식으로 학습됩니다.

왜 한 번의 Forward Pass로 충분한가?

텍스트가 n개의 토큰이면, n번 모델을 돌려야 하는 것 아닌가? 아닙니다! Transformer의 구조 덕분에 한 번이면 됩니다.

핵심 질문: 학습도 안 했는데 왜 "330" 뒤에서 <API> 확률이 높은가?

이것은 ToolFormer를 이해하는 데 가장 중요한 질문입니다.
맞습니다 — 이 시점에서 fine-tuning은 아직 하지 않았습니다. 그렇다면 대체 어떻게?

In-Context Learning이 확률을 만드는 과정 — 시각화

프롬프트의 예시들이 어떻게 Attention 패턴을 형성하여 특정 위치에서 API 호출 확률을 높이는지 단계별로 봅니다.

Step B 시각화: 위치별 확률 히트맵

예시 문장을 선택하면, 각 토큰 위치에서 <API> 토큰이 나올 확률을 색상 강도로 보여줍니다. 진한 색 = 높은 확률 = "여기서 도구를 쓰고 싶다!"

높은 확률

Step C: 후보 생성 — 완전한 워크스루

확률이 높은 위치(top-k)를 선택한 후, 각 위치에서 LLM이 API 호출 텍스트를 자동으로 생성합니다. 아래 시뮬레이션은 하나의 문장이 전체 샘플링 과정을 거치는 모습을 보여줍니다.

각 API별 독립 샘플링

중요한 점: 5가지 API에 대해 각각 독립적으로 이 과정을 수행합니다. 같은 텍스트라도 각 API의 관점에서 다른 위치가 선택됩니다.

"The population of Seoul is approximately 9.7 million as of 2023."

Top-k 위치 선택 & m개 샘플링

각 위치 i에서 p_i = P_M(<API> | prompt(x), x₁, ..., x_i)

→ 확률이 높은 상위 k개 위치를 선택 (논문: k=5)
→ 각 위치에서 m개의 API 호출을 샘플링 (논문: m=5)

총 후보 수 = k × m = 5 × 5 = 25개 후보/텍스트 (API당)

왜 같은 위치에서 m개를 뽑나?

같은 위치, 다른 API 호출 (m=3 예시)

위치: "...is ▮ 330 meters tall..."

후보 1: [QA("height of Eiffel Tower")]
후보 2: [QA("how tall is the Eiffel Tower")]
후보 3: [QA("Eiffel Tower height in meters")]

→ LLM이 temperature 샘플링으로 다양한 표현 생성
→ 나중에 필터링에서 가장 유용한 것만 살아남음

왜 다양성이 중요한가

후보 1: [QA("Eiffel Tower height")]
→ 결과: "330 meters" ✓ 유용!

후보 2: [QA("who built the Eiffel Tower")]
→ 결과: "Gustave Eiffel" ✗ 이 위치에선 불필요

→ 같은 질문이라도 표현에 따라 API 결과가 달라짐
→ 여러 후보 중 최고를 고르기 위해 다양하게 생성

전체 샘플링 과정 — 한 문장의 여정

Q&A: 샘플링에 대한 핵심 질문들

Q: LLM이 API 호출 형식을 어떻게 아나요? 학습한 적이 없는데? ▼
Few-shot 프롬프트가 핵심입니다! LLM의 in-context learning 능력을 활용합니다. 프롬프트에 "[Calculator(2+3)]" 같은 예시를 2~5개 보여주면, LLM은 패턴을 파악하고 새로운 텍스트에서도 같은 형식으로 생성할 수 있습니다.

이것은 GPT-3의 few-shot learning 능력과 동일한 원리입니다. 별도의 학습 없이, 프롬프트만으로 새로운 형식을 따라할 수 있습니다.
Q: 모든 위치에서 확률을 계산하면 너무 느리지 않나요? ▼
LLM의 forward pass 한 번으로 모든 위치의 확률을 동시에 구할 수 있습니다! Transformer는 teacher forcing으로 한 번에 모든 토큰의 다음 토큰 분포를 계산하므로, 각 위치에서 "<API>" 토큰의 확률을 추출하기만 하면 됩니다.

비용이 많이 드는 부분은 top-k 위치에서 API 호출 텍스트를 생성(디코딩)하는 단계입니다. 이래서 k와 m을 적절히 제한합니다 (k=5, m=5).
Q: 프롬프트 예시 2~5개를 사람이 만드는 건 "사람 개입"이 아닌가요? ▼
핵심적인 차이가 있습니다:

기존 방법: 수천~수만 개의 (텍스트, 도구 호출) 쌍을 사람이 직접 annotation
ToolFormer: API "사용법"만 5개 예시로 설명. 마치 매뉴얼을 한 번 건네주는 것

비유하자면: 사람에게 "계산기 사용법"을 설명하는 것 vs "이 100개 수학 문제에서 계산기를 어디에 사용해야 하는지 일일이 표시"하는 것의 차이입니다. ToolFormer는 전자만 필요합니다.
Q: 잘못된 API 호출 (엉뚱한 질문, 잘못된 계산식)도 생성되나요? ▼
네, 당연히 생성됩니다! 그래서 필터링 단계가 존재합니다.

예를 들어 "The weather is nice today"에서 [Calculator(nice + today)] 같은 의미 없는 호출이 생성될 수 있습니다. 하지만 이런 호출은:
① API 실행 자체가 실패하거나
② 결과가 나와도 후속 텍스트 예측에 전혀 도움이 안 되므로
→ Loss 기반 필터링에서 자동 제거됩니다.

샘플링은 "많이 생성"하고, 필터링이 "엄격히 선별"합니다.

5. Step 2: 유용성 필터링 (핵심!)

필터링 기준: Weighted Cross-Entropy Loss

API 호출이 진짜 도움이 되는지를 수학적으로 판단합니다.

τ_i = min( L_i⁺ , L_i )   에 대해 필터링 조건:

L_i⁻  = API 호출 없이 나머지 텍스트를 예측하는 loss
L_i⁺  = API 호출 + 결과를 넣었을 때 나머지 예측 loss
L_i  = API 호출만 넣고 결과 없이 예측하는 loss

필터 조건: L_i⁻ − τ_i ≥ threshold

즉, "API를 안 쓸 때의 loss" − "API를 쓸 때의 loss" ≥ threshold
→ API가 충분히 loss를 줄여야 살아남음!

시각적 예시: 어떤 API 호출이 살아남나?

필터링의 직관적 의미

✓ 살아남는 경우

"이 건물은 [QA("에펠탑의 높이")] → 330m 높이의..."

→ API 없이는 "300m", "320m" 등 불확실
→ API 결과로 정확히 "330m" 예측 가능
→ Loss가 크게 감소! ✓

✗ 필터링되는 경우

"오늘 날씨가 [QA("날씨란 무엇인가")] → 대기의 상태... 좋아서..."

→ API 없이도 "좋아서" 예측 가능
→ API 결과가 예측에 도움 안 됨
→ Loss 변화 없음/증가 ✗

6. Step 3: Fine-tuning

데이터셋 구성 (C* 생성)

필터링을 통과한 API 호출들을 원본 텍스트에 삽입하여 augmented 데이터셋 C*를 만듭니다.

원본 (C):
Joe Biden은 미국의 46대 대통령이다. 그는 1942년 11월 20일에 태어났다.

Augmented (C*):
Joe Biden은 [QA("미국 46대 대통령")]→ Joe Biden 미국의 46대 대통령이다. 그는 [Calendar()]→ Today is Feb 9, 2023 [Calculator(2023 - 1942)]→ 81 1942년 11월 20일에 태어났다.

학습 목표: 표준 Language Modeling Loss

L(θ) = −Σ log P(x_i | x_1, ..., x_{i-1} ; θ)

단, 텍스트 x에는 [API_call → result] 토큰도 포함됨

9. 실험 결과

벤치마크 성능 비교

ToolFormer (GPT-J 6.7B)가 훨씬 큰 모델들과 비교한 결과:

핵심 발견

1. 도구 사용은 선택적

모든 문장에서 도구를 쓰지 않습니다. 모델이 불확실할 때만 도구를 호출합니다. 필요 없을 때는 도구 없이 생성합니다.

2. 작은 모델의 역전

6.7B ToolFormer가 175B GPT-3를 일부 태스크에서 능가! 도구가 모델 크기의 한계를 보완합니다.

3. 언어 모델링 유지

도구 학습 후에도 일반 언어 모델링 능력(perplexity)이 유지됩니다. 도구를 배우면서 기존 능력을 잃지 않습니다.

4. Self-supervised의 위력

사람의 annotation 없이, LM의 자체 신호만으로 도구 사용을 학습. 새로운 도구 추가도 쉽습니다.

한계점

  • 도구를 체인으로 사용하는 것 (한 도구의 출력을 다른 도구의 입력으로)은 아직 제한적
  • 인터랙티브 도구 (검색 후 클릭 등)는 지원 불가
  • 샘플링 데이터셋(CCNet)의 품질에 영향을 받음
  • 더 큰 base 모델로의 확장은 아직 실험되지 않음

요약: ToolFormer의 전체 흐름

전체 과정을 한 그림으로
① 데이터 준비
CCNet 코퍼스에서 텍스트 가져옴
↓
② Few-shot 프롬프팅
각 위치에서 API 호출 후보 생성 (k×m개)
↓
③ API 실행
후보 API 호출을 실제로 실행하여 결과 획득
↓
④ 필터링
L⁻ − min(L⁺, L) ≥ τ 조건으로 유용한 것만 선별
↓
⑤ 데이터 Augmentation
통과한 API 호출을 원본 텍스트에 삽입 → C*
↓
⑥ Fine-tuning
C*로 LM을 fine-tune → ToolFormer 완성!
