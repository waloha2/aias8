0OVERVIEW

RAG는 LLM에 "기억과 자료실"을 붙이는 일

LLM은 학습 시점에 본 것만 안다. 회사 문서, 최신 뉴스, 사내 위키는 모른다. RAG(Retrieval-Augmented Generation)는 질문이 들어올 때마다 관련 문서를 찾아서 LLM에게 함께 보여주는 방식이다. 즉, "전공 시험을 오픈북으로 푸는 학생"을 만들어주는 기술이다.

전체 파이프라인 — 인덱싱(Indexing) & 질의(Query)의 두 단계
인덱싱 단계 (오프라인) 질의 단계 (온라인) 데이터 흐름
🎯

왜 RAG인가

LLM은 학습 데이터 이후의 정보를 모르고, 사내 문서·고객 데이터·정책처럼 비공개 자료에 접근할 수 없다. Fine-tuning보다 빠르고 싸며, 문서를 바꾸면 즉시 답변에 반영된다.

🧱

두 단계로 나뉜다

① 인덱싱은 무거운 한 번 작업 — 문서를 잘게 자르고 벡터로 변환해 DB에 저장. ② 질의는 가벼운 실시간 작업 — 검색 후 LLM에 컨텍스트로 주입.

⚖️

품질의 80%는 검색

LLM 모델만 좋다고 RAG가 좋아지지 않는다. 검색 단계에서 엉뚱한 청크가 올라오면 LLM은 자신 있게 틀린 답을 만든다. 임베딩 · 청킹 · 검색 단계가 핵심이다.

1EMBEDDING MODELS

임베딩 — 텍스트를 "의미의 좌표"로 바꾸는 일

임베딩은 텍스트를 수백~수천 차원의 숫자 벡터로 바꾸는 함수다. 의미가 비슷한 문장은 벡터 공간에서 가깝게 놓이도록 학습되어 있다. RAG의 검색 품질은 이 모델의 성능에 크게 의존한다.

개념도 — 텍스트를 벡터 공간의 점으로

주요 임베딩 모델 비교

최신 벤치마크 기반의 일반적 경향이며, 실제 도메인·언어에 따라 결과가 달라진다. 본인의 데이터로 직접 평가하는 것이 정석.

🇰🇷 한국어 RAG에서 반드시 챙길 것

1. "다국어" ≠ "한국어 잘함"

영어 MTEB 1위가 한국어에서도 1위라는 보장은 없다. 실측 결과 OpenAI text-embedding-3-large는 한국어에서 bge-m3보다 약 7%p 낮다 (0.617 vs 0.687). SFR·gte-Qwen2 같은 7B 거대 모델조차 한국어에선 bge-m3보다 떨어진다.

2. 토큰화 — 한 글자가 1토큰이 아니다

한국어는 BPE 기준 토큰이 영어보다 1.3~2배 많이 든다. 같은 청크 크기여도 한국어가 더 많은 토큰을 소비. 가격·context 길이 계산 시 고려.

3. 도메인 어휘 — 의료/법률/특허

한국 의료/법률/특허 코퍼스는 일반 학습 데이터에 적게 포함되어 있다. 도메인 특화 fine-tuning(KURE, ko-sroberta)이나 BGE-M3 + 도메인 어휘 보강을 고려.

4. 한·영 혼합 문서 처리

실제 한국 기업 문서엔 영어 약어, 영문 제품명, 영문 인용이 섞여 있다. multilingual 모델 또는 bge-m3 같이 cross-lingual에 강한 모델이 안전.

5. 정규화 & 전처리

조사/어미가 검색 품질에 영향을 줄 수 있다. 다만 최근 임베딩 모델은 이를 잘 흡수하므로 형태소 분석은 sparse(BM25) 쪽에서만 활용하고 dense 임베딩엔 원문 그대로 넣는 것이 보통.

6. 평가 — 본인 데이터로 직접

Ko-MTEB·KoBEST 같은 공개 벤치는 시작점일 뿐. 실제 운영 쿼리 50~100개를 만들어 Recall@5, MRR을 측정해 모델을 골라야 한다.

💡현실적인 출발점 (한국어 RAG)
  • 최우선 후보: nlpai-lab/KURE-v1 — 공개 모델 중 한국어 검색 1위 (NDCG@10 0.695). 무료, 자체 호스팅, bge-m3 기반.
  • 안정적인 베이스라인: BAAI/bge-m3 — 한국어 외 100+ 언어 지원. KURE-v1과 차이는 ~1%p로 미미.
  • API 옵션: voyage-3-large — 한국어 단독 벤치는 부재하지만 다국어에서 강력. 자체 평가셋으로 검증 권장.
  • OpenAI 선호 시: text-embedding-3-large는 한국어에서 위 모델들보다 7%p 뒤처지므로, 한국어가 주력이면 비추천.
2CHUNKING

청킹 — 문서를 어떻게 자를 것인가

LLM은 한 번에 받을 수 있는 텍스트 길이에 한계가 있고, 임베딩 모델도 너무 긴 텍스트는 의미를 흐릿하게 표현한다. 그래서 긴 문서를 의미 있는 조각(chunk)으로 나눈다. 청킹은 RAG 품질에서 가장 저평가된 변수다 — 검색이 잘 안 되는 이유의 절반은 청킹이 잘못된 탓이다.

너무 작으면

맥락이 사라진다. "그 사람"이 누구를 가리키는지, 표 제목이 무엇이었는지 모른다. 검색은 되어도 LLM이 답을 못 만든다.

적당하면

하나의 청크 = 하나의 자기 완결적 사실. 임베딩이 의미를 또렷이 잡고, LLM이 그대로 인용할 수 있다.

너무 크면

여러 주제가 한 벡터에 평균화 되어 검색이 둔해진다. 컨텍스트 윈도우와 토큰 비용도 폭증.

청킹 전략 5가지

1

Fixed-size Chunking 가장 단순

고정 글자수(또는 토큰수)로 자른다. 일반적으로 chunk_size=500~1000, overlap=50~200.

구현 LangChainCharacterTextSplitter LangChainTokenTextSplitter LlamaIndexTokenTextSplitter LlamaIndexSentenceSplitter
RAG는 검색 증강 생성의 약자다. LLM에 외부 지식을 주입해 정확한 답변을 생성하게 한다. 임베딩 모델은 텍스트를 벡터로 변환한다. 벡터는 의미가 비슷할수록 가깝게 배치된다. 청킹은 긴 문서를 적절한 크기로 자르는 작업이다. 너무 크면 검색이 둔해지고 너무 작으면 맥락이 사라진다.

👍 장점

  • 구현이 즉시 가능 — text[i:i+500] 한 줄로 끝
  • 예측 가능한 토큰 비용 — 모든 청크 정확히 500토큰, 인보이스 계산 쉬움
  • 벡터 DB 처리량 일정 — 임베딩 batch 크기·메모리 사용량이 일정

👎 단점

  • 문장 중간에서 잘림 — 예: "환불은 14일 이내에 가능하며" / "결제 수단으로 환불됩니다"로 분리되어 둘 다 정답 못 됨
  • 의미 단위 무시 — "Q3 매출은 12% 성장" 다음 문장 "비용은 8% 증가"가 다른 청크로 가서 LLM이 둘을 연결 못함
  • 표/코드/리스트가 깨짐 — Markdown 표가 중간에 잘려 헤더 행과 데이터 행이 따로 검색됨

언제 쓰나? 빠른 프로토타입, 길이가 비슷한 동질 문서(예: 트윗·SMS 모음), 베이스라인 측정용.

2

Recursive Chunking 실무 기본값

구분자 우선순위를 정해 큰 단위(문단 → 문장 → 단어)로 시도하며 크기를 맞춘다. 코드(Python/JS/…)는 언어별 separator를 자동 적용해주는 변종을 쓰면 함수·클래스 경계를 잘 보존.

구현 LangChainRecursiveCharacterTextSplitter LangChainRecursiveCharacterTextSplitter.from_language() LlamaIndexSentenceSplitter LlamaIndexCodeSplitter

👍 장점

  • 자연스러운 경계에서 절단 — 문단(\n\n) → 문장(. ! ?) → 단어 순으로 시도, 가능한 큰 단위 보존
  • 크기 상한 보장 — chunk_size 넘어가면 강제로 자르므로 LLM context 폭주 위험 없음
  • 대부분 도메인에서 합리적 — Chroma 벤치마크 Recall 0.881로 베이스라인 강자
  • 코드는 언어별 separator — Python은 class · def, JS는 function · const를 우선순위로

👎 단점

  • 의미를 이해하진 않음 — "1.2 절차" 다음 "1.3 예외사항"이 한 청크에 묶이면 주제가 섞임 (문법만 보고 자르므로)
  • 표/코드는 여전히 깨질 수 있음 — Markdown 표가 chunk_size 초과시 행 중간에서 잘림
  • 대명사·암묵 주어 문제 — "그 회사의 매출은…" 청크에서 '그 회사'가 누구인지 모름

언제 쓰나? 처음부터 의심 없이 시작할 수 있는 기본값. 90%의 일반 텍스트(블로그·뉴스·매뉴얼)에 충분.

3

Semantic Chunking 의미 기반

문장을 임베딩 → 인접 문장 간 의미 거리 측정 → 거리가 갑자기 커지는 지점에서 자른다. 주제가 바뀌는 곳을 자동 감지.

구현 LangChainlangchain_experimental.text_splitter.SemanticChunker LlamaIndexSemanticSplitterNodeParser Chroma 연구ClusterSemanticChunker · LLMSemanticChunker

의미 거리가 임계선을 넘는 지점에서 청크 경계가 만들어진다

👍 장점

  • 주제 단위로 자연스럽게 분리 — 에세이에서 "도입부 → 본론1 → 본론2 → 결론" 같은 구획이 헤딩 없어도 자동 검출됨
  • 가변 길이 — 한 주제가 길면 청크도 크게(2000자), 짧으면 작게(200자). 고정 사이즈의 경직성 해결

👎 단점

  • 인덱싱 시 임베딩 비용 증가 — 100만 문장 코퍼스라면 청킹 한 번에 100만 번 임베딩(인덱싱 시간 10~30배)
  • 임계값 튜닝 필요 — percentile 90/95/99 등을 직접 시도해봐야 함. 데이터마다 최적이 다름
  • 짧은 문서에선 효과 미미 — 트윗·짧은 FAQ는 어차피 한 주제 → 의미 거리 계산이 노이즈만
  • Chroma 벤치마크에서 Recall 0.847 — Recursive(0.881)보다 오히려 낮은 경우도

언제 쓰나? 긴 에세이·논문·리포트처럼 한 문서 안에 주제가 여럿일 때. 단, "Semantic = 항상 더 좋다"는 아님 — 베이스라인 측정 후 결정.

SEMANTIC CHUNKING의 3가지 변종

"문장 사이 의미 거리로 자른다"는 같은 원리에서 출발하지만, 어떻게 거리를 재고 어디서 자르는지가 다르다. Chroma 벤치마크에 등장하는 3가지를 정리.

전통 방식

KamradtSemanticChunker

Greg Kamradt가 LangChain 5 Levels of Splitting에서 제안한 원조. 인접 문장 임베딩의 코사인 거리를 측정하고, 거리가 percentile 임계값(예: 95%)을 넘는 지점에서 자른다. 그리디(greedy) 알고리즘.

Chroma Recall 0.847 — 의외로 Recursive보다 낮음.

📦 LangChain SemanticChunker 기본 동작.

Chroma 신작

ClusterSemanticChunker

Chroma Research가 2024년 발표. 문서를 먼저 50토큰 단위로 자른 뒤, 동적 프로그래밍으로 "한 청크 안의 문장 쌍 코사인 유사도 합을 최대화"하는 전역 최적 그룹핑을 찾는다. 탐욕적이지 않고 globally optimal.

Chroma Recall 0.913 (max 400 tokens) — 임베딩만으로 거의 최상위.

📦 chunking_evaluation 패키지에 구현.

LLM 호출

LLMSemanticChunker

문서를 한 번에 LLM에게 보여주고 "의미 단위 경계가 어디인지" 직접 물어본다. 예: "이 텍스트를 의미가 일관된 청크로 나누고, 각 청크의 시작·끝 인덱스를 JSON으로 알려줘."

Chroma Recall 0.919 (최고) — but 전체 corpus를 LLM에 통과시키므로 비용 ↑↑↑.

📦 chunking_evaluation 패키지 · 대용량은 비현실적.

Structural Chunking 문서 구조 활용

Markdown 헤딩, HTML 태그, PDF 섹션, 코드의 함수/클래스 같은 문서 자체의 구조를 경계로 삼는다.

구현 LangChainMarkdownHeaderTextSplitter LangChainHTMLHeaderTextSplitter LangChainMarkdownTextSplitter LlamaIndexMarkdownNodeParser · HTMLNodeParser LlamaIndexHierarchicalNodeParser PDF/Layoutunstructured · Docling · LlamaParse
# 1. 환불 정책 (h1) 회사는 구매 후 14일 이내 환불을 허용한다. ## 1.1 적용 대상 (h2) 온라인 스토어에서 구매한 모든 제품. 단, 소프트웨어 라이선스 제외. ## 1.2 환불 절차 (h2) 1. 주문 페이지에서 "환불 요청" 클릭 2. 사유 작성 후 제출 3. 영업일 기준 3~5일 이내 처리 # 2. 교환 정책 (h1) 교환은 동일 제품의 다른 옵션(색상/사이즈)만 가능하다.

👍 장점

  • 의미 경계가 명확 — "## 1.1 환불 절차" 헤더 아래 내용이 통째로 한 청크. 저자가 의도한 단위 그대로
  • 각 청크가 자기 완결적 — "이 섹션은 무엇에 관한 것인지" 헤더로 알 수 있음. 대명사 문제 적음
  • 섹션 제목을 메타데이터로 활용 — 검색 시 { "section": "1.1 환불 절차" }로 필터링·인용 가능

👎 단점

  • 구조가 없는 문서엔 적용 불가 — 스캔 PDF, 메모, 채팅 로그엔 헤딩이 없음
  • 섹션마다 크기 편차 큼 — "1.1 개요"는 200자, "1.5 상세 절차"는 5000자 → 임베딩 품질 불균일
  • 파서별 구현 비용 — PDF 헤더 검출은 layout 모델(예: unstructured hi_res) 필요, 정확도 90% 정도

언제 쓰나? 사내 위키, API 문서, 약관, 매뉴얼, 논문, 코드베이스. Markdown/HTML/AsciiDoc 등 구조가 명확한 포맷.

STRUCTURAL CHUNKING의 변종 — PDF에서 자주 쓰는 두 가지

"문서 구조를 따른다"는 같은 철학에서, PDF·보고서처럼 페이지·섹션 경계가 명확한 문서엔 다음 두 가지가 표준이 된다. NVIDIA 벤치마크에서 둘 다 비교 대상으로 등장.

NVIDIA 평균 1위

📄 Page-level Chunking

PDF의 각 페이지 = 한 청크로 그대로 사용. 가장 자연스러운 경계 중 하나로, 페이지가 보통 하나의 시각적·논리적 단위로 디자인되어 있음을 이용. 표·그림·캡션이 페이지 안에 함께 갇혀 있어 맥락 보존이 좋다.

  • 인용·출처가 "p.34"로 명확 → 사용자 신뢰↑
  • 구현 단순 — 페이지 추출만 하면 됨
  • 경계가 정해져 있어 다양한 문서에서 안정적 (std 0.107)

NVIDIA Answer Accuracy 0.648 (1위) · 5개 데이터셋 평균.

📦 NVIDIA NeMo Retriever, PyMuPDF.get_text() per page, unstructured.partition_pdf(strategy="hi_res").

조심해서 사용

📑 Section-level Chunking

섹션 헤더(예: 1.1, 1.2, 2.) 단위로 청크 생성. PDF 레이아웃 모델로 헤딩을 검출해야 한다. Markdown Header Splitting의 PDF 버전이라고 보면 된다.

  • 섹션 크기 편차 큼 — 1.1은 1문단, 1.2는 10페이지일 수 있음
  • 헤딩 검출 정확도가 변수 (스캔 PDF·복잡 레이아웃에서 취약)
  • NVIDIA 실험에서 Page-level보다 평균 낮음 (FinanceBench만 예외)

"섹션이 곧 의미 단위"라는 직관과 달리, 페이지 단위가 실측 더 안정적.

📦 NVIDIA nemoretriever-parse, unstructured + heading filter, Docling.

💡 관계 정리 — Structural은 우산 개념(헤딩 기반 일반). Page-level과 Section-level은 PDF에 특화된 두 가지 구체 구현. Markdown에서 MarkdownHeaderTextSplitter를 쓰는 것이 곧 "Section-level"이고, PDF의 시각적 페이지를 그대로 자르는 것이 "Page-level"이다.
5

Contextual Chunking Anthropic 권장

각 청크에 "이 조각이 어디서 왔고 어떤 문맥인지" 한 줄 요약을 LLM으로 자동 생성해서 앞에 붙이는 기법. Anthropic이 발표한 Contextual Retrieval 기법의 핵심. 검색 실패율을 최대 49%까지 줄였다는 보고.

구현 Anthropic Cookbookcontextual-embeddings (custom prompt + prompt caching) LlamaIndexllama_index.packs.contextual_retrieval LangChaincustom RunnableLambda + ChatPromptTemplate ※ 주의LangChain에 전용 클래스 없음 — 직접 prompt 체인 구성

👍 장점

  • 고립된 청크의 모호함을 제거 — "매출은 12% 성장" → "ACME 2024 Q3 북미 SaaS 사업부 매출은 12% 성장"으로 자동 보강
  • 대명사·암묵 주어 문제 해결 — "그 회사는…", "위 조항에 따라…" 같은 표현이 다 자기 완결적이 됨
  • 검색 정확도 큰 폭 향상 — Anthropic 실험: retrieval 실패율 최대 49% 감소 (BM25 + reranker와 결합 시)

👎 단점

  • 인덱싱 시 LLM 호출 비용 — 청크마다 1번씩 LLM 호출. 100만 청크 = 100만 호출 (Haiku 기준 ~$2~5, GPT-4 기준 수십~수백 달러)
  • prompt caching으로만 현실적 — 같은 문서 전체를 매번 보내야 하므로 캐싱 없으면 비용 폭증. Anthropic API의 prompt caching 90% 할인이 사실상 필수
  • 저장 용량 증가 — 청크 앞에 50~150토큰 컨텍스트 prefix가 붙음. 인덱스 크기 10~30% ↑
  • 실시간 갱신 어려움 — 문서 일부가 바뀌면 영향받는 청크의 context를 다 재생성

언제 쓰나? 청크가 짧고 문맥 의존이 큰 경우(재무 보고서, 법률 조항, 채팅 로그). 비용을 감당할 수 있다면 거의 모든 경우 추천.

📊 청킹 벤치마크 — 실제 측정 결과

"어떤 청킹이 가장 좋은가"는 오랫동안 감으로만 답해 왔다. 2024~2025년 들어 체계적인 평가 논문이 여럿 나왔다. 두 개의 대표 연구를 본다.

BENCHMARK 1

Retrieval-level 평가 · Recall · Precision · IoU

"Evaluating Chunking Strategies for Retrieval" — 청킹 자체의 검색 품질을 측정. 문서를 청킹하고 검색기로 청크를 가져왔을 때, 그 청크에 정답 텍스트가 얼마나 포함되어 있는지를 Recall(빠뜨리지 않음), Precision Ω(노이즈 적음), IoU(정답 영역과 겹치는 정도)로 측정.

Recursive 400자/overlap 0이 의외로 강하다 (0.881). 비싼 Semantic 청킹 없이도 베이스라인을 훌쩍 넘는다. "복잡한 청킹이 항상 좋다"는 통념을 깬다.

💡 발견 2

LLM 기반 의미 청킹이 진짜 1위. 다만 모든 sentence를 LLM에 통과시켜야 해서 인덱싱 비용이 수십~수백 배 든다.

💡 발견 3

Overlap의 효과는 미미하다. Recall은 +1%p, Precision은 -3%p. 비싼 트레이드오프. 0~10% 정도면 충분.

💡 발견 1 — Page-level의 부상

페이지 단위 청킹이 평균 1위(0.648)이자 가장 안정적(std 0.107). 자연스러운 경계 + 인용 시 페이지 번호로 출처 추적 용이. PDF·보고서 위주라면 페이지 청킹을 베이스라인으로.

💡 발견 2 — 극단값은 손해

128토큰은 너무 짧고(맥락 손실), 2048토큰은 너무 길다(노이즈 증가). KG-RAG에서 128토큰은 0.421로 최하위, RAGBattlePacket에서 2048은 0.749 (1024의 0.804 대비 감소).

💡 발견 3 — 쿼리가 청크 크기를 결정

같은 데이터셋이라도 "값을 묻는 질문"엔 256~512, "관계·이유를 묻는 질문"엔 1024 또는 페이지. 한 RAG 안에 두 타입이 섞이면 Hybrid Chunking + Query Routing 고려.

두 벤치마크의 공통 결론 — 실무가 챙겨야 할 4가지

1. "기본값"부터 잘 골라라

Chroma: RecursiveCharacterTextSplitter 400자/overlap 0 → NVIDIA: Page-level 또는 Token 1024. 출발점만 잘 골라도 베이스라인 90% 이상은 잡힌다.

2. Overlap은 절제해서

Chroma: overlap 200 시 Recall +1%p, Precision -3%p.
NVIDIA: 10/15/20% 비교 시 15%가 sweet spot. 50% 같은 과한 overlap은 토큰 비용만 부풀린다.

3. Semantic 청킹은 케이스 별 판단

전통적 임베딩 기반 Semantic 청킹은 Recursive보다 살짝 낮을 수 있다(Chroma: 0.847 vs 0.881). LLM 기반 Semantic만 명확히 우월(0.919)하지만 비용이 수십 배.

4. 한 방법으로 다 끝나지 않는다

NVIDIA에서 같은 "재무 문서"라도 데이터셋에 따라 최적이 Page / Token 512 / Token 1024로 갈렸다. 3개 정도 후보를 실제 평가셋으로 비교하는 것이 정답.

3VECTOR DATABASE

벡터 DB — 수백만 개의 벡터를 빠르게 검색

벡터 DB는 "코사인 유사도가 가장 높은 K개를 찾아 줘"를 밀리초 단위로 답해주는 데이터베이스다. 내부적으로는 ANN(Approximate Nearest Neighbor) 인덱스(HNSW, IVF, ScaNN 등)를 사용한다. 모든 벡터를 일일이 비교하면(brute force) 100만 개에 수 초가 걸리지만, ANN으로는 수 밀리초로 줄어든다.

ANN 직관 — 모든 점을 비교하지 않고 "근처"부터 빠르게 탐색

상층에서 대략적으로 이동 → 하층에서 정밀 탐색. log(N) 시간으로 검색 가능.

  • 벤더 벤치마크를 그대로 믿음 — Qdrant 벤치는 Qdrant 1위, TigerData는 pgvector 1위. 본인 워크로드로 측정해야.
  • "벡터 DB만 바꾸면 검색이 좋아진다" — 임베딩 모델/청킹이 retrieval 품질의 진짜 변수. DB는 속도·운영 문제.
  • HNSW 파라미터 디폴트만 씀 — M(연결도, 8~72), ef_construct(64~512), ef_search(8~768) 튜닝으로 5~10배 차이 가능.
  • recall 측정 없이 latency만 비교 — 어떤 엔진은 recall 80%로 떨어뜨려 빠른 속도 보고. 항상 동일 recall 목표에서 비교.
  • 인덱스가 RAM에 안 들어가는데도 RAM 사이즈만큼 데이터 쌓음 — 디스크로 spill되면 latency 10~100배. 메모리 예산 미리 계산.
  • 메타데이터 필터를 사후(post)로만 적용 — Top-K가 비어버림. pre-filter 지원하는 DB 선택 (Qdrant, pgvector with btree).
4RETRIEVAL

검색 방식 — Dense, Sparse, 그리고 Hybrid

좋은 RAG의 핵심은 "사용자 질문에 진짜로 답이 들어 있는 청크를 Top-K에 올리는" 일이다. 여기에는 의미 기반(Dense)과 키워드 기반(Sparse) 두 패러다임이 있고, 둘을 합친 Hybrid가 실무 정답에 가깝다.

두 검색의 차이 — 같은 질문, 다른 결과

세 가지 검색 방식 깊게 보기

🧬 Dense Retrieval

질문과 문서를 같은 임베딩 모델로 벡터화한 뒤, 코사인 유사도 / 내적으로 가까운 청크를 찾는다. 이 강의 1번 섹션이 다룬 그 방식.

강점

  • 동의어·문맥적 의미를 잘 잡는다
  • "환불"으로 "반품/취소" 문서를 찾을 수 있다
  • 크로스 링귀얼(한→영) 가능

약점

  • 고유명사·제품 코드·약어가 약하다 (학습 데이터에 없으면)
  • 드문 단어(롱테일)에 둔감
  • 임베딩 비용·GPU 의존

🔤 Sparse Retrieval — BM25

고전적인 정보 검색 알고리즘. 문서 내 단어 빈도(TF), 전체 코퍼스에서 단어의 희소성(IDF), 문서 길이 보정으로 점수를 매긴다. 학습 없이도 동작.

BM25 점수 직관: 질문에 있는 단어가 청크에 자주 나오면서, 그 단어가 코퍼스 전체에선 드물수록, 점수↑.
예) "401K" 같은 희귀 단어 1번 = "정책" 5번보다 강하다.

강점

  • 정확한 단어/숫자/코드 검색에 최강
  • 학습 불필요, 즉시 작동
  • 해석 가능 (왜 매칭됐는지 명확)
  • CPU만으로도 빠름

약점

  • 동의어 모름 ("자동차" vs "차량")
  • 한국어는 형태소 분석 필요 ("환불을" vs "환불")
  • 문맥·의도 이해 불가

한국어 팁: Mecab/Kkma/Okt로 형태소 분석 후 BM25를 돌리면 효과가 크게 오른다. SPLADE 같은 학습형 sparse 모델도 옵션.

🔀 Hybrid Retrieval — 두 검색을 합친다

Dense와 Sparse를 동시에 돌리고 결과를 합친다. 가장 흔한 합치는 법은 Reciprocal Rank Fusion (RRF).

score(d) = Σ retriever r 1 / (k + rankr(d))

k는 보통 60. 어떤 검색기에서든 상위에 들면 점수 가산.

문서 Dense 순위 Sparse 순위 RRF 점수 최종 순위
"환불 정책 v3" 2 1 1/62 + 1/61 = 0.0328 🥇 1위
"반품 절차 안내" 1 5 1/61 + 1/65 = 0.0318 🥈 2위
"지불 수단 종류" 8 2 1/68 + 1/62 = 0.0308 🥉 3위

두 신호 모두에 강한 문서가 가장 위로 올라온다. 한 쪽만 잘 잡힌 문서도 잃지 않는다.

권장 검색 파이프라인 — Hybrid + Reranker

Top-50씩 가져와 RRF로 합치고, Cross-encoder Reranker로 정밀 재정렬 → 최종 3~10개

🎯 Reranker란?

검색기는 빠르지만 거칠다. Reranker는 (질문, 후보 청크)를 한 쌍씩 LLM-style 모델에 넣어 정밀히 점수를 매긴다. Top-50을 Top-5로 줄일 때 마지막 정확도 부스터.

📐 Top-K는 몇이 좋을까

검색 단계는 넉넉하게(50~100), Reranker 후 LLM에 들어가는 건 짧게(3~10). 너무 많으면 LLM이 산만해지고 비용이 오른다 (Lost-in-the-middle 현상).

🔬 평가 지표

Recall@K (관련 문서가 Top-K에 들었나), MRR (관련 문서가 얼마나 위에 있나), nDCG (순위 가중). 프로덕션에선 LLM-as-a-judge로 답변 품질도 함께 측정.

5PROMPT DESIGN

프롬프트 설계 — LLM이 검색 결과로 좋은 답을 만들게 하기

좋은 청크가 검색됐어도 프롬프트가 엉성하면 LLM은 자기가 학습한 일반 지식으로 답해 버린다. 프롬프트는 LLM에게 "이 자료만 보고, 근거를 달고, 모르면 모른다고 해라"는 세 가지 규칙을 명확히 강제해야 한다.

BAD 😵 (자주 보는 안티패턴)
아래 문서를 참고해서 사용자 질문에 답해 줘. 문서: {context} 질문: {question} 답변: // 문제 1 — 구분자 없음: LLM이 문서와 질문 경계를 흐려서 봄 // 문제 2 — 출처 인용 강제 없음: 환각 발견 어려움 // 문제 3 — "모르면 모른다" 규칙 없음: LLM이 알아서 지어냄 // 문제 4 — 역할/페르소나 없음: 답 톤이 일관되지 않음
GOOD 🎯 (3원칙 적용)
[역할] 당신은 회사 정책 문서를 기반으로 답하는 사내 어시스턴트입니다. [규칙] 1. 반드시 아래 <context> 안의 정보만 사용하세요. 2. 답변 끝에 사용한 문서 번호 [doc_id]를 대괄호로 인용하세요. 3. context에 답이 없으면 정확히 "제공된 자료에서 확인할 수 없습니다" 라고만 답하세요. 추측 금지. <context> [doc_1] 환불은 구매 후 14일 이내… [doc_2] 디지털 상품은 환불 불가… </context> [질문] {question} [답변]
📥

1. 컨텍스트 주입

검색된 청크를 명확한 구분자(예: <context> 태그)로 감싼다. 각 청크에 [doc_1] 같은 ID를 붙여 LLM이 인용할 수 있게 한다. 시스템 프롬프트와 사용자 메시지를 명확히 분리.

📎

2. Citation 강제

답변에 [doc_id]를 붙이라고 명시적으로 지시. 사용자가 근거를 클릭해 원문을 확인할 수 있어야 신뢰가 생긴다. Citation 없이는 RAG가 아니라 그냥 LLM 답변일 뿐.

🚫

3. "모르면 모른다"

검색 결과에 답이 없을 때 LLM은 환각(hallucination)을 일으킨다. "context에 없으면 '확인할 수 없습니다'라고만 답하라"는 문장 한 줄이 환각의 70~80%를 막는다. 예외 문구는 정확히 한 가지로 고정.

💬 잘 작동하는 출력 예시

모델 출력 (User Facing)Q. 디지털 상품도 환불되나요?
디지털 상품(소프트웨어 라이선스, 이북 등)은 환불 대상에서 제외됩니다 [doc_2]. 일반 상품은 구매 후 14일 이내에 환불 신청이 가능합니다 [doc_1]. ──────────────────────── 📎 인용된 문서: [doc_1] 환불 정책 v3 · 1.1 적용 대상 [doc_2] 환불 정책 v3 · 1.3 제외 항목

🛡️ 환각 방지 패턴 더 보기

패턴 A — "Cite or refuse"

규칙: 모든 사실 주장에는 [doc_id]
인용을 반드시 붙여야 한다.
인용할 수 있는 근거가 없으면 답하지 마라.

강하게 강제하면 답변이 짧아지지만, 확신 없는 정보를 토하지 않는다.

패턴 B — 신뢰도 자기 평가

답변 후, 신뢰도를 다음 중 하나로 표기:
- 확실 (context에 직접 명시)
- 추정 (context로 추론)
- 불확실 (context에 부분 정보만)

사용자가 답변의 무게를 직접 판단할 수 있다.

패턴 C — 충돌 시 명시

두 문서가 서로 다른 내용을 말하면
어느 쪽도 추측하지 말고
"문서 [doc_1]은 X, [doc_2]는 Y"
라고 그대로 보여줘라.

정책 충돌·버전 차이를 사용자에게 그대로 노출.

패턴 D — 출처 검증 단계

최종 답변 전에 다음을 자문:
1) 내가 쓴 모든 사실이 context에 있나?
2) context에 없는 표현이 있나?
3) [doc_id] 인용이 빠진 문장이 있나?

Chain-of-verification 형태. 더 큰 모델일수록 효과 큼.

⚡ 한 줄 요약 — 프롬프트의 일은 LLM의 "지식"이 아니라 "근면성"을 강제하는 것이다. "주어진 자료만 보고, 인용하고, 모르면 모른다고 해라" — 이 세 가지면 RAG의 답변 신뢰도가 극적으로 오른다.

개념 이해

  • RAG의 두 단계(인덱싱 / 질의)를 그림 없이 설명할 수 있다

  • 청킹 5가지 전략의 차이를 한 줄로 말할 수 있다
  • ANN(HNSW)이 왜 brute force보다 빠른지 직관이 있다
  • Dense vs Sparse vs Hybrid의 차이를 예시로 설명할 수 있다
  • Citation과 "모름" 처리가 왜 중요한지 안다

실전 출발 키트

  • 임베딩: KURE-v1(한국어 1위) 또는 bge-m3
  • 청킹: Recursive, 800자/100자 overlap
  • 벡터 DB: Chroma(PoC) → Qdrant/pgvector(프로덕션)
  • 검색: Dense Top-50 + BM25 Top-50 → RRF → Reranker
  • 프롬프트: 3원칙(context · citation · "모름")
  • 평가셋 50~100 쿼리로 Recall@5 측정 시작

1 RAG의 가장 본질적인 효과 — 다음 중 RAG가 fine-tuning과 가장 다르게 잘 해결하는 문제는?
  • A. LLM이 출력하는 말투·페르소나를 일관되게 만든다
  • B. 학습 시점 이후의 새 사실이나 사내 자료를 답변에 반영한다
  • C. 모델의 추론 능력 자체를 향상시킨다
  • D. 한국어 문법 오류를 줄인다
  • E. 모델 파라미터 수를 줄여서 비용을 낮춘다
정답 보기
정답: B RAG는 "학습 시점 이후의 정보 / 학습 데이터에 없던 자료"를 검색으로 주입하는 패러다임. 말투(A)·추론능력(C)·문법(D)·파라미터(E)는 RAG가 직접 다루는 문제가 아니며, 말투·추론은 fine-tuning이, 비용은 모델 선택·캐싱이 다룬다. A는 fine-tuning과 잘 어울리고, RAG와는 보완재로 같이 쓰는 경우가 많다.
2 임베딩 모델 교체에 대한 다음 설명 중 옳은 것은?
  • A. 임베딩 모델만 바꾸면 되고, 기존 벡터는 그대로 쓸 수 있다
  • B. 임베딩 모델을 바꾸면 모든 청크의 벡터를 다시 만들어야 한다
  • C. 임베딩 모델은 검색에만 영향을 주고, 답변 품질에는 영향이 없다
  • D. 다국어 모델은 영어 단독 모델보다 항상 영어 검색 성능도 더 좋다
정답 보기
정답: B 모델마다 벡터 공간이 다르기 때문에 새 모델 ↔ 옛 벡터의 거리는 무의미. 교체 시 인덱스 전체를 다시 구축해야 하므로 임베딩 선택은 처음에 신중히. D는 흔한 오해 — 영어 단독 모델이 영어 작업에서는 더 자주 우위. C는 검색이 나빠지면 LLM이 잘못된 청크로 답하므로 결과가 연쇄적으로 나빠진다.

  • A. Fixed-size 400자
  • B. Fixed-size 1200자
  • C. Recursive (LangChain식)
  • D. Sentence-aware (~800자)
  • E. Semantic (BGE merge)
정답 보기
정답: D (Sentence-aware, 50.6%) C(Recursive 50.4%)와 거의 동률 1·2위, B(1200자) 49.2%, E(Semantic) 48.2%, A(400자) 46.4%. "경계를 깨지 않는다"는 원칙이 핵심. Fixed-small은 청크에 multi-hop 사실이 다 안 들어와 손해, Semantic은 뉴스 토픽 변화가 잦아 오히려 사실을 분리.
4 "한 청크에 너무 많은 토픽이 섞이면 임베딩이 흐려진다" — 이 말의 가장 정확한 의미는?
  • A. 임베딩 모델이 청크를 처리 못해 에러가 발생한다
  • B. 청크 벡터가 여러 토픽의 평균이 되어 어느 토픽 질문에도 1등으로 잡히지 않는다
  • C. 벡터 차원이 자동으로 줄어든다
  • D. BM25 점수가 항상 낮아진다
정답 보기
정답: B Embedding이 토픽들의 평균 위치로 떨어져, 특정 토픽 질문 벡터와의 cosine이 어중간해진다. 그 결과 작은 청크 여러 개에 토픽이 분리돼 있을 때보다 retrieval recall이 떨어진다. A는 사실이 아님(임베딩은 안 깨짐), D는 BM25는 토픽이 아니라 단어 매칭이라 별개.
5 Hybrid 검색(Dense + BM25 + RRF) 도입 시 가장 비용 효율적인 시점은?
  • A. 모델을 더 큰 것으로 바꾼 후
  • B. RAG 시스템을 처음 만들 때부터
  • C. 사용자 트래픽이 급증한 후
  • D. 도메인 fine-tuning을 끝낸 후
정답 보기
정답: B RRF는 코드 10줄 수준, 추가 비용은 BM25 인덱스 한 벌(매우 저렴). 본 강의 파일럿에서 dense 37.8% → hybrid 50.6%로 +12.8pt 무료 점프. 나중에 도입하기보다 처음부터 default로 가는 것이 항상 이득. A·C·D는 모두 hybrid 도입을 미룰 이유가 되지 않는다.
6 OX. RRF(Reciprocal Rank Fusion) 공식에서 점수 자체가 아니라 순위를 합치는 이유는, Dense의 cosine 점수(0~1)와 BM25의 점수(분포가 corpus마다 다름)가 스케일이 달라 직접 더하면 한쪽이 항상 이기기 때문이다.
  • A. ⭕ 맞다 (True)
  • B. ❌ 틀리다 (False)
정답 보기
정답: A (True) BM25 점수는 corpus 통계에 따라 0~수십까지 분포가 자유롭고, dense cosine은 보통 0~1. 직접 합치면 BM25 우세. 순위로 바꾸면 두 검색이 동일 스케일(1~N)이 되어 자연 결합된다. k=60은 Cormack et al. 2009의 default. 30~100 사이에서 도메인에 맞춰 조정.

  • C. "Error code TS-999 의미"
  • D. "이 정책의 의도가 뭐야?"
정답 보기
정답: C 고유 식별자·코드명은 동의어가 없고 정확한 문자열 일치가 답. BM25는 이 매칭에 압도적, dense는 의미 공간상 비슷한 다른 코드도 끌어와 노이즈를 만든다. A·B·D는 의역·동의어가 풍부해 dense가 잘 하는 영역. 실무에서는 Hybrid로 두 신호 모두 활용하는 것이 정답.
